Day 2 — Async Message Queues — RabbitMQ + Celery
What is a message queue and why does it matter for microservices?
In Day 1 you saw synchronous service communication — Service A calls Service B and waits for the response. This works for fast operations but breaks down for slow ones:

Without message queue (sync):
User uploads 100MB PDF
→ API waits for chunking (30s) + embedding (20s) + storing (5s)
→ User stares at loading spinner for 55 seconds
→ If connection drops → everything lost
→ If ingestion service crashes → user gets 500 error

With message queue (async):
User uploads 100MB PDF
→ API puts "process this document" message in queue → returns INSTANTLY
→ Worker picks up message → processes in background
→ User sees "processing..." immediately
→ If connection drops → message is safe in queue
→ If worker crashes → message stays in queue → another worker picks it up
=================================================================================================================
The mental model
Producer (FastAPI) → Queue (RabbitMQ) → Consumer (Celery Worker)

Producer:  "Here's a job — do this whenever you can"
Queue:     "Got it — I'll hold this safely until a worker is free"
Consumer:  "I'm ready — give me the next job"

Key properties:
Durability:   Messages survive queue restarts
Persistence:  Messages survive worker crashes
Acknowledgment: Worker confirms completion before message is deleted
Retry:        Failed messages go back to queue automatically
==================================================================================================================
RabbitMQ vs Celery — what's the difference?
RabbitMQ:
→ The message broker — stores and routes messages
→ A server you run (like PostgreSQL for databases)
→ Knows nothing about Python or your code
→ Just receives, holds, and delivers messages

Celery:
→ Python task queue library
→ Uses RabbitMQ (or Redis) as its broker
→ Knows about Python functions — turns them into tasks
→ Manages workers, retries, schedules
→ The part you write Python code for

Analogy:
RabbitMQ = postal service (infrastructure)
Celery    = the sender and recipient (your code)

Topic 2 — Celery Basics → 02_celery_basics.py
What is Celery and why does it sit on top of RabbitMQ?
RabbitMQ is infrastructure — it knows nothing about Python. Celery is the Python layer that turns your regular functions into distributed tasks.

RabbitMQ is the road system (infrastructure that moves messages), 
while Celery is the Domino's Pizza delivery company (Python application that uses those roads to deliver tasks, track orders, 
handle failures, and report results) — Celery sits on top of RabbitMQ because RabbitMQ provides reliable message transport, 
and Celery provides the Python-friendly task management that makes distributed computing practical.

Topic 3 — Celery + FastAPI Integration → 03_celery_fastapi.py
The core pattern
FastAPI handles HTTP requests and dispatches work to Celery. The client gets an immediate response with a task_id, then polls for results.
Client → POST /documents/upload
         ↓
FastAPI validates + saves file
FastAPI → celery_app.delay() → RabbitMQ queue
FastAPI → returns {"task_id": "abc-123", "status": "processing"}

Client → GET /tasks/abc-123 (polls every 2 seconds)
         ↓ status: "processing" (still in queue)
         ↓ status: "processing" (worker picked it up)
         ↓ status: "success" (done!)
